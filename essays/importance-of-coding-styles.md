I spend a lot of time concered about the consistency of my code. Not just if it works or not, but if it looks nice or if the semicolons are all spaced the same in all of my for-loops, or if I type "x++" or "++x." I don't do this for any particular reason. I just do it because I'm not happy if something is off. When I write papers, I spend more time adjusting the margins than writing a paragraph.

That being said, I've never considered whether these concepts are important in coding. I was taught this in ICS111, so I will code this way for a reason that I don't even know. I know that it looks nice, and that's all that matters.

But after using ESLint and being enforced coding standards, I've developed some opinions about them, some good and some bad.

## The Positives

Given that I already try to enforce coding standards on myself without using ESLint, using it wasn't that bad. It's like having my own coding conscience inside of IntelliJ yelling at me when I don't put a space between a function name and the parameters. Sure it adds a little extra time on the clock, but at a low cost and a potentially positive outcome.

Outside of that, given the importance of open source in the coding community, coding standards are almost a no-brainer. It improves the readability of code, and can make multiple different programmers code look the same. When multiple people collaborate on the same project, rather than a jumble of different formats, it would look like it's been written all by the same neat person. After reflecting on SMART questions and the importance of them in the open source and coding community, it's important to not waste anyone's time, and coding standards make it easier to not waste time at all.

Along with the improvement of readability, comes the ability to detect errors and bugs quicker. A messy code is like a messy room. It's hard to find what you need at any given time. But when it's all cleaned up, it's a lot easier to spot the bug. This essentially increases the productivity of coding, at the minimal cost of cleaning up the code in the process.

## The Negatives

When I said that I developed opinions about coding standards, they weren't all going to be positive. Sure there are a lot of benefits to coding standards, but at the same time, it comes with some consequences.

I've realized to a certain extent, when writing code for the past year, I've developed my own coding standard on really small things, which most of the time, ESLint calls me out on. I understand the importance of uniformity when coding, but who's to say that ESLint is the "correct" way. Sure, it's used by a lot of other coders, but there are other coding standards too. Maybe in the future, it's replaced by another standard. It seems like a "trend" in a sense of the current era.

Now even though it does help in finding bugs, it doesn't actually prevent bugs. It's like enforcing unnecessary action on aesthetic over function. Code that looks nice makes no difference on the product than code that doesn't follow coding standards. Different languages have their own coding standards as well, and that's a whole other thing.

## Overall...

These are really just opinions. Most of them are probably widespread opinions, but opinions nonetheless. It is true the coding standards make no overall change to the resulting code, but they help in the long term. Despite the negatives, I do believe that coding standards are necessary, but that's just personal preference. I don't see them as a massively important thing, but I enjoy their benefits. They can be a pain sometimes, but things don't have to be absolutely good with no downsides to be beneficial.
